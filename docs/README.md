## طريقة كتابة التوثيق

كل مهمة تنفيذية يجب أن تُوثّق في مكانين فقط:

1. ملف الخطة: `tasks/10-production-readiness-plan.md`
2. ملف الجلسة: `../.kilo/sessions/10-production-readiness.md`

ويُحدَّث `tasks/master-task-list.md` فقط لتغيير الحالة أو إضافة مرجع للملفات النشطة.

لا تُنشأ ملفات تقارير جديدة باسم `task*` أو ملفات جلسة داخل `docs/`. ملفات الجلسات تكون داخل:

```txt
.kilo/sessions/
```

## مبدأ الاكتمال العمودي

لا نعمل على كامل المشروع بنسبة `10/100` في كل جزء. كل مرحلة يجب أن تنتهي عمودياً:

1. تحديد فجوة حقيقية.
2. تنفيذ الحد الأدنى المنتج.
3. إضافة الاختبارات حسب الحاجة.
4. تحديث التوثيق.
5. تشغيل التحقق المناسب.
6. إغلاق المرحلة قبل الانتقال إلى غيرها.

## حدود التعقيد المطلوبة

- الواجهة الأمامية:
  - الكود واضح وقابل للتعديل.
  - لا يوجد over-engineering في UI.
  - حالات الخطأ والنجاح والتحميل تُضاف فقط حيث تؤثر على تجربة المستخدم.
  - الاختبارات تُكتب للمكونات والflows الحرجة، وليس لكل مكون.

- طبقة العميل:
  - هدفها الأساسي خدمة UX و no-env behavior.
  - لا تُعامل كطبقة أمان لأنها قابلة للتعديل في المتصفح.
  - لا توضع فيها قواعد أمان تعتمد على إخفاء أو منع المستخدم.

- طبقة الخادم / Supabase:
  - هي الطبقة الأهم للأمان والبيانات.
  - RLS والمigrations والـ Edge Functions تحتاج اختبارات وتوثيق أقوى.
  - لا تُعتمد على rate limit أو authorization في المتصفح فقط.

- التوثيق:
  - يجب أن يوضح مخطط الملفات والمجلدات.
  - يجب أن يحتوي على diagram عند وجود بنية أو تدفق مهم.
  - يجب أن يفرّق بين الحالة الحالية وخطة الإنتاج.
